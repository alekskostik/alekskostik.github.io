Стандарт
процесса разработки

Единый стандарт команды — от постановки задачи до промышленной эксплуатации. Правила процесса соблюдаются так же строго, как соглашение по стилю кода.

Об этом стандарте
Стандарт проверен в продуктовой и проектной разработке и объединяет опыт разных команд. Цель — понятный, обоснованный путь, охватывающий все критичные циклы жизни разработки. Может и должен быть адаптирован под конкретную команду с учётом её ресурсов и компетенций: отдельные этапы допустимо упрощать или объединять.

01Постановка
задачи
Бриф Формулировка проблемы Стейкхолдеры Бизнес-цели
Заказчик или менеджер формулирует проблему, бизнес-цели и ожидаемый результат. Идентифицируются все заинтересованные стороны. Фиксируется согласованный бриф. Ответственный: PM / Владелец продукта.
02Анализ
AS-IS TO-BE GAP-анализ Карта рисков
Аналитик и архитектор изучают текущее состояние, требования, ограничения и интеграции. Проводятся интервью со стейкхолдерами, мэппинг процессов. Ответственный: Аналитик / Архитектор.
03ЧТЗ
Функц. требования Роли пользователей Критерии приёмки Согласование
Частное техническое задание: описывает что делает система, понятным для заказчика языком. Версионируется, проходит ревью и подписывается всеми сторонами. Ответственный: Аналитик.
04ОТР и ПМИ
HLD (опц.) Архитектура Спецификация API Модель данных Тест-план Тест-кейсы
Описание технического решения отвечает на вопрос как система реализована. Для сложных проектов рекомендуется предварять ОТР кратким HLD (High-Level Design) — 1–3 страницы схем и ключевых решений, которые согласуются с командой до написания детального ОТР. ПМИ разрабатывается параллельно — QA участвует в ревью ОТР и одновременно готовит программу испытаний. Разработка не начинается без утверждённых ПМИ. Ответственные: Архитектор / Tech Lead + QA Lead.
05Декомпозиция
на эпики
Эпики Пользовательские истории Дорожная карта
На основе ЧТЗ и ОТР менеджер вместе с командой разбивает работу на эпики — крупные блоки с измеримой бизнес-ценностью. Каждый эпик соответствует разделу ЧТЗ. Ответственный: PM + команда.
06Планирование
спринтов
Бэклог спринта Оценки задач Критерии готовности Демо заказчику Ретроспектива
Эпики декомпозируются на задачи, оцениваются и распределяются по спринтам (2 недели). Церемонии спринта: планирование (в начале), дейли (ежедневно), демо заказчику (в конце спринта), ретроспектива (сразу после демо — разбор что шло хорошо, что улучшить, конкретные договорённости на следующий спринт). Ответственный: Scrum-мастер + команда.
07Разработка
Feature-ветка Код + unit-тесты Запрос на слияние (MR) Код-ревью CI: сборка + тесты Деплой в DEV
Каждая задача реализуется в отдельной feature-ветке от основной ветки разработки. По завершении открывается запрос на слияние (MR) в GitLab — с описанием изменений и ссылкой на задачу в Yougile. MR проходит код-ревью минимум одним другим разработчиком и CI-проверку (сборка + автотесты). Мерж без зелёного CI и без апрува ревьюера запрещён. После мержа — автодеплой в DEV-среду. Ответственный: разработчик + ревьюер.
08Тестирование
Модульные тесты Системное тестирование Регрессия Приёмочное (НАТ/UAT) Отчёт о тестировании
Тестирование — явная самостоятельная фаза, не часть разработки. После фиксации функциональности QA проводит системное и регрессионное тестирование в TEST-среде по ПМИ. Критические и высокие дефекты блокируют переход в следующую среду. Ответственные: QA + разработчики.
09Деплой по
средам
DEV TEST CERT PROD Примечания к релизу План отката
Движение строго последовательное — каждый переход требует контрольной точки:
DEV → TEST ← системное тестирование QA
TEST → CERT ← 0 критических/высоких дефектов
CERT → PROD  ← акт приёмки + план отката
Ответственные: DevOps + PM + QA.
10ОПЭ
План ОПЭ Мониторинг 24/7 Журнал инцидентов Акт завершения ОПЭ
Опытно-промышленная эксплуатация — работа системы в PROD с усиленным мониторингом командой разработки. Срок: 1–4 недели (фиксируется в плане работ или ЧТЗ). Все инциденты в журнале. Без подписанного акта завершения ОПЭ проект не считается сданным. Ответственные: PM + Dev + Заказчик.
11Промышленная
эксплуатация
Акт приёмки SLA Эксплуатационная документация Инструкция администратора
Система передаётся заказчику или эксплуатационной команде. Подписан акт приёмки, зафиксировано SLA, передана полная документация. Команда разработки переходит в режим поддержки согласно утверждённому плану поддержки. Ответственные: PM + DevOps.
Р-01

Постановка задачи

ПРАВИЛО-001
Нет проблемы — нет задачи
Любая задача начинается с формулировки проблемы. Без чёткого ответа «какую боль мы решаем?» задача не берётся в работу.
✓ ПРАВИЛЬНО«Время поиска документа превышает 3 мин — пользователи теряют задачи»
✗ НЕПРАВИЛЬНО«Нужно переделать поиск — заказчик так хочет»
ПРАВИЛО-002
Все стейкхолдеры определены до старта
Все заинтересованные стороны идентифицированы и приглашены к согласованию ещё на этапе брифа. Узнавать о новых участниках в процессе работы — недопустимо.
ПРАВИЛО-003
Бриф подписывается до перехода к анализу
Фаза анализа не начинается без согласованного брифа. Даже если задача кажется очевидной — договорённости фиксируются письменно.
Р-02

Документация

ПРАВИЛО-010
ЧТЗ — истина для бизнеса, ОТР — истина для разработки
Разграничение строгое. ЧТЗ описывает ЧТО делает система. ОТР описывает КАК она это делает. Смешивать нельзя.
✓ В ЧТЗ«Пользователь может экспортировать отчёт в PDF»
✗ В ЧТЗ«Для экспорта используем библиотеку pdfmake v4»
ПРАВИЛО-011
Версионирование обязательно
Все документы (ЧТЗ, ОТР, ПМИ) версионируются. Каждое изменение после согласования — новая версия с описанием правок. Хранение в Git или Confluence.
ПРАВИЛО-012
ПМИ пишется параллельно с ОТР
QA участвует в ревью ОТР и одновременно готовит тест-план. Нельзя начинать разработку без утверждённых критериев приёмки.
ПРАВИЛО-013
Критерии приёмки должны быть зафиксированы до начала разработки
Критерии приёмки для каждого требования обязательны — но могут находиться либо в ЧТЗ (рядом с требованием), либо в ПМИ (в матрице тест-кейсов). Главное — они должны быть согласованы до старта разработки. Без зафиксированных критериев нельзя однозначно определить, выполнено ли требование.
✓ ДОПУСТИМОКритерии в ЧТЗ § 6.2 или в ПМИ как тест-кейсы — при условии, что оба документа готовы до спринта
✗ НЕПРАВИЛЬНО«Критерии определим по ходу» — или их нет ни в ЧТЗ, ни в ПМИ
Р-03

Декомпозиция и планирование

ПРАВИЛО-020
Эпик = ценность для бизнеса
Каждый эпик имеет явную бизнес-ценность и соответствует разделу ЧТЗ. Эпик без измеримого результата — не эпик.
ПРАВИЛО-021
Задача — от 2 часов до 3 дней
Задача не меньше 2 ч и не больше 3 дней. Золотое правило: одна задача — один запрос на слияние кода.
✓ ПРАВИЛЬНО«Реализовать эндпоинт POST /api/documents» (4 ч)
✗ НЕПРАВИЛЬНО«Реализовать модуль документов» (2 нед.)
ПРАВИЛО-022
Состав спринта неприкосновенен
После планирования состав спринта не меняется без решения команды. Новые задачи — в следующий спринт. Исключение: критический сбой в продуктиве.
ПРАВИЛО-023
Нет задачи без связи с эпиком
Каждая задача привязана к эпику. «Висячих» задач не бывает. Технический долг — отдельный эпик, а не скрытые задачи.
Р-04

Разработка и код-ревью

ПРАВИЛО-025
Каждая задача — отдельная feature-ветка
Разработка ведётся в изолированных ветках, создаваемых от основной ветки разработки. Имя ветки содержит номер задачи из Yougile. Работа напрямую в основной ветке запрещена.
✓ ПРАВИЛЬНОfeature/YGL-123-auth-login
✗ НЕПРАВИЛЬНОКоммит напрямую в main или develop
ПРАВИЛО-026
MR обязателен — мерж без ревью запрещён
Любое изменение в основную ветку проходит через запрос на слияние (MR) в GitLab. MR содержит: описание изменений, ссылку на задачу в Yougile, скриншоты или видео при изменении UI. Мерж без апрува ревьюера и зелёного CI невозможен технически (защита ветки в GitLab).
ПРАВИЛО-027
Код-ревью — не дольше 1 рабочего дня
Открытый MR должен быть рассмотрен в течение 1 рабочего дня. Ревьюер проверяет: соответствие задаче, читаемость, потенциальные ошибки, покрытие тестами. Замечания формулируются конкретно и без личной оценки автора.
✓ ПРАВИЛЬНО«Эта функция не обрабатывает пустой список — добавь проверку»
✗ НЕПРАВИЛЬНО«Плохой код» / MR висит без ревью 3 дня
ПРАВИЛО-028
CI должен быть зелёным перед мержем
CI-пайплайн (сборка + автотесты) обязан завершиться успешно до мержа MR. Мерж с падающим CI запрещён даже при наличии апрува. Если CI упал по инфраструктурной причине — решение принимается техлидом явно.
Р-05

Процессные правила

ПРАВИЛО-030
Прослеживаемость: задача → история → эпик → ЧТЗ
По любой задаче в трекере можно дойти до требования в ЧТЗ. Обратное тоже верно: у каждого требования есть задача.
ПРАВИЛО-031
Изменение требований — только через запрос на изменение
Изменение требований после согласования ЧТЗ оформляется как запрос на изменение: оценка влияния, согласование сторонами, обновление документов.
ПРАВИЛО-032
Демо — обязательный артефакт каждого спринта
Каждый спринт заканчивается демонстрацией заказчику. Нет демо — нет закрытия спринта. Обратная связь фиксируется и идёт в бэклог.
ПРАВИЛО-033
Ретроспектива — обязательная церемония, а не формальность
Ретроспектива проводится после каждого демо. Формат: что шло хорошо, что мешало, конкретные договорённости на следующий спринт. Каждый пункт «улучшить» должен иметь ответственного и срок. Выводы фиксируются письменно (GitLab Wiki / Yougile).
✓ ПРАВИЛЬНО«Договорились: код-ревью не дольше 1 дня — проверит PM, следующий спринт»
✗ НЕПРАВИЛЬНО«Поговорили, всё норм» — без записи и ответственных
Р-06

Тестирование

ПРАВИЛО-040
Тестирование — отдельная явная фаза
QA — не часть «разработки» в бэклоге. Системное тестирование проводится в TEST-среде после фиксации функциональности по тест-кейсам ПМИ.
✓ ПРАВИЛЬНОФиксация → QA в TEST → Отчёт → переход в CERT
✗ НЕПРАВИЛЬНО«QA смотрит по ходу, сразу деплоим в PROD»
ПРАВИЛО-041
Все уровни тестирования обязательны
Модульные тесты — ответственность разработчика (в спринте). Системное и регрессионное — QA (TEST). Приёмочное — с участием заказчика (CERT). Пропустить уровень — только с письменным обоснованием PM.
ПРАВИЛО-042
Критические и высокие дефекты блокируют переход
Ни один критический или высокий дефект не переносится в следующий релиз. Переход TEST→CERT и CERT→PROD — только при 0 открытых критических/высоких дефектах.
ПРАВИЛО-043
Отчёт о тестировании обязателен перед релизом
Перед деплоем в CERT или PROD QA выпускает отчёт: покрытие, пройдено / провалено / заблокировано, статус дефектов. Без отчёта деплой не согласовывается.
Р-07

Среды и деплой

ПРАВИЛО-050
Строгая последовательность: DEV → TEST → CERT → PROD
Деплой через среду запрещён. Каждый переход — явная контрольная точка: результаты тестирования, подпись ответственного, примечания к релизу.
✓ ПРАВИЛЬНОDEV → TEST (gate QA) → CERT (gate UAT) → PROD (акт)
✗ НЕПРАВИЛЬНО«Хотфикс напрямую в PROD, протестируем потом»
ПРАВИЛО-051
План отката обязателен перед каждым PROD-деплоем
До деплоя в PROD описан и проверен план отката: процедура, ответственный, максимальное время выполнения. Тестируется в CERT.
ПРАВИЛО-052
CERT-среда максимально близка к PROD
Конфигурация, анонимизированные данные и инфраструктура CERT зеркалируют PROD. Тестирование на «облегчённой» среде не считается полноценным приёмочным испытанием.
Р-08

Опытно-промышленная эксплуатация

ПРАВИЛО-060
Параметры ОПЭ фиксируются в проектной документации
Период ОПЭ, его длительность, критерии успешного завершения и состав команды поддержки оговариваются до начала разработки и фиксируются в плане работ или ЧТЗ.
ПРАВИЛО-061
Во время ОПЭ — усиленная поддержка
Мониторинг 24/7 с алертами. Время реакции на критический инцидент ≤ 4 ч, на высокий ≤ 1 рабочего дня. Все инциденты фиксируются в журнале и разбираются.
ПРАВИЛО-062
Акт завершения ОПЭ — условие сдачи проекта
Без подписанного акта завершения ОПЭ проект не считается сданным. Акт содержит: количество инцидентов, их статус, заключение заказчика о соответствии ЧТЗ.
✓ ЗАВЕРШЕНИЕАкт ОПЭ → передача в эксплуатацию → утверждённый план поддержки / SLA
✗ НЕПРАВИЛЬНО«Деплоим и расходимся, остальное доделаем потом»
Р-09

Примечания к релизу

ПРАВИЛО-070
Release Notes готовятся перед каждым деплоем в PROD
Примечания к релизу — обязательный артефакт любого деплоя в PROD. Составляются на основе задач, отмеченных тегом релиза в Yougile. Публикуются в GitLab Wiki до деплоя.
✓ ПРАВИЛЬНОЗадачи в Yougile отмечены тегом релиза → собираем список → публикуем в wiki до деплоя
✗ НЕПРАВИЛЬНО«Что изменилось — смотрите коммиты в GitLab»
ПРАВИЛО-071
Тегирование задач к релизу обязательно
Каждая задача, попадающая в релиз, помечается тегом версии в Yougile (например, v1.3.0) до начала финального тестирования. Задачи без тега в релиз не включаются.
ПРАВИЛО-072
Два уровня детализации: для команды и для заказчика
Для внутренней команды и тестировщиков — полный список изменений с ссылками на задачи. Для заказчика — краткая выжимка на понятном языке, только для крупных или значимых релизов. Оба варианта хранятся в GitLab Wiki.
Р-10

Разбор инцидентов (пост-мортем)

ПРАВИЛО-080
Пост-мортем проводится после каждого значимого инцидента
Поводы для пост-мортема: критический инцидент в PROD или ОПЭ, значимый сбой в процессе разработки (потеря данных в TEST, сорванный спринт по техническим причинам). Срок проведения — не позднее 3 рабочих дней после инцидента, пока детали свежи.
ПРАВИЛО-081
Пост-мортем — это анализ, а не поиск виноватых
Главный принцип: культура без обвинений (blameless). Цель — понять, что пошло не так в процессе или системе, а не кто виноват. Участники говорят открыто. Выводы фиксируются и доступны всей команде в GitLab Wiki.
✓ ПРАВИЛЬНО«Алертинг не сработал, потому что порог был выставлен слишком высоко — исправляем порог»
✗ НЕПРАВИЛЬНО«Разработчик не проверил — впредь будьте внимательнее»
ПРАВИЛО-082
Каждый пост-мортем завершается конкретными действиями
Без списка корректирующих действий пост-мортем не считается завершённым. Каждое действие — задача в Yougile с ответственным и сроком. Статус задач проверяется на следующей ретроспективе.
Иерархия декомпозиции

ЧТЗ
Частное техническое задание. Описывает функциональные требования конкретной системы или модуля с точки зрения бизнеса и пользователя. Язык доступен заказчику. Отвечает на вопрос «что?». Не содержит технических решений.

Отличие от ТЗ: ЧТЗ — это детализация ТЗ для конкретной подсистемы или функционального блока. ТЗ задаёт общие рамки проекта, ЧТЗ — конкретные требования к отдельной части. В небольших проектах ЧТЗ может заменять ТЗ целиком.
ТЗ
Техническое задание. Документ верхнего уровня, определяющий общие цели, рамки и требования к проекту в целом. Является основой для разработки ЧТЗ на отдельные подсистемы.

Отличие от ЧТЗ: ТЗ охватывает проект целиком и может оставаться на высоком уровне абстракции. ЧТЗ — детальный документ по конкретному функциональному блоку, на основе которого ведётся разработка. В практике команды основным рабочим документом является ЧТЗ.
ОТР
Описание технического решения. Описывает архитектуру, стек, API, модель данных и технические решения. Язык технический. Отвечает на вопрос «как?». Ссылается на ЧТЗ.
ПМИ
Программа и методика испытаний. Определяет что, как, кем и когда проверяется. Содержит тест-кейсы, критерии приёмки и форму акта. Разрабатывается параллельно с ОТР.
ОПЭ
Опытно-промышленная эксплуатация. Ограниченный период работы системы в PROD с усиленным мониторингом команды (1–4 нед.). Завершается актом — без него проект не считается сданным.
Эпик
Крупный блок функциональности с измеримой бизнес-ценностью. Соответствует разделу ЧТЗ. Декомпозируется на истории. Длительность разработки: 4–8 недель.
Пользовательская история
Описание функции: «Как [роль], я хочу [действие], чтобы [цель]». Обязательно имеет критерии приёмки. Декомпозируется на задачи. Входит в состав эпика.
Спринт
Итерация разработки, 2 недели. Включает: планирование → ежедневные встречи → демо заказчику → ретроспектива. Результат — рабочий инкремент, принятый QA.
Критерии готовности
Согласованный список условий, при выполнении которых задача считается завершённой. Например: код написан, ревью пройдено, тесты зелёные, принято QA.
Критерии приёмки
Условия, при которых требование или история считается реализованной. Формат: «Дано / Когда / Тогда». Без критериев приёмки задача не берётся в спринт.
Запрос на изменение
Формализованная процедура изменения согласованных требований. Включает: оценку влияния на сроки и бюджет, согласование сторонами, обновление документов.
Прослеживаемость
Наличие связи: задача → история → эпик → требование ЧТЗ. По любой задаче можно дойти до требования и обратно. Обеспечивает полноту покрытия разработки.
ADR (Architecture Decision Record)
Запись об архитектурном решении. Короткий структурированный документ: контекст (почему встал вопрос), рассмотренные варианты, принятое решение, последствия. Хранится в ОТР или wiki. Никогда не удаляется — при смене решения старый ADR помечается устаревшим, новый добавляется рядом. Позволяет команде понимать, почему система устроена именно так.
НАТ
Натурные (приёмочные) испытания с участием представителей заказчика. Проводятся в CERT-среде по ПМИ. Результат — подписанный акт приёмки. Синоним UAT.
Среды: DEV / TEST / CERT / PROD
DEV — разработка. TEST — системное тестирование QA. CERT — приёмочное тестирование, зеркало PROD. PROD — промышленная среда. Переход строго последовательный через контрольные точки.
Фиксация функциональности
Момент остановки разработки новых возможностей перед системным тестированием. После этого момента допускается только исправление дефектов, найденных QA.
UAT (User Acceptance Testing) — приёмочное тестирование
Тестирование с участием заказчика или конечных пользователей в CERT-среде по тест-кейсам ПМИ. Цель — подтвердить соответствие системы ЧТЗ в реальных условиях.
План отката
Задокументированная и протестированная процедура возврата к предыдущей версии при неудачном деплое. Включает: шаги, ответственного, максимальное время. Тестируется в CERT.
Контрольная точка (Gate)
Явная процедура согласования перехода между средами: результаты тестирования, статус дефектов, подпись ответственного. Без прохождения gate деплой в следующую среду запрещён.
Технический долг
Накопленные решения, снижающие качество кода или архитектуры. Оформляется как отдельный эпик в бэклоге — явно, с оценкой и приоритизацией. Никогда не скрывается в других задачах.
Бриф
Краткий согласованный документ, фиксирующий проблему, бизнес-цели, ожидаемый результат и стейкхолдеров. Входной артефакт для фазы анализа. Подписывается до старта работ.
HLD (High-Level Design)
Высокоуровневое описание архитектуры: ключевые компоненты, их взаимодействие, выбор стека — без детализации реализации. Создаётся между ЧТЗ и полным ОТР. Может быть вводным разделом ОТР или отдельным документом. Особенно полезен для сложных проектов, где нужно быстро согласовать архитектурный подход.
Критерии приёмки — где размещать?
Могут быть зафиксированы либо в ЧТЗ (рядом с каждым требованием) — типично для Agile, либо в ПМИ (в виде тест-кейсов) — типично для проектного подхода. Оба варианта допустимы. Главное правило: критерии должны существовать и быть согласованы до начала разработки требования.
Ретроспектива
Командная встреча в конце каждого спринта. Три вопроса: что шло хорошо, что мешало, что конкретно изменим. Каждый пункт «изменить» получает ответственного и срок. Результаты фиксируются письменно. Цель — непрерывное улучшение процесса, а не разбор полётов ради формальности.
Release Notes (примечания к релизу)
Документ, описывающий что изменилось в конкретной версии системы. Составляется на основе задач с тегом релиза в Yougile. Существует в двух версиях: полной — для команды и тестировщиков (со ссылками на задачи), краткой — для заказчика (на понятном языке, только для крупных релизов). Хранится в GitLab Wiki.
Пост-мортем
Структурированный разбор значимого инцидента или сбоя. Цель — понять первопричину и предотвратить повторение, а не найти виноватых (принцип blameless). Проводится не позднее 3 дней после инцидента. Результат — список конкретных задач в Yougile. Публикуется в GitLab Wiki и доступен всей команде.
Blameless (культура без обвинений)
Принцип разбора инцидентов: причины ищутся в процессах и системах, а не в ошибках конкретного человека. Люди допускают ошибки — задача команды выстроить процесс так, чтобы одна ошибка не приводила к сбою. Без этого принципа пост-мортемы превращаются в формальность.
RCA (анализ первопричины)
Root Cause Analysis — метод поиска глубинной причины инцидента. Вопрос «почему?» задаётся несколько раз подряд (метод «5 почему»), пока не найдена системная причина, а не симптом. Используется в пост-мортемах.
MR (Merge Request)
Запрос на слияние кода в GitLab. Основной инструмент код-ревью и контроля качества. Содержит описание изменений и ссылку на задачу в Yougile. Мерж возможен только при наличии апрува ревьюера и зелёного CI. Аналог Pull Request в других системах.
Feature-ветка
Изолированная ветка в GitLab для разработки одной задачи. Создаётся от основной ветки разработки, имя содержит номер задачи. После завершения работы сливается в основную ветку через MR. Прямые коммиты в основную ветку запрещены.
CI (непрерывная интеграция)
Continuous Integration — автоматический запуск сборки и тестов при каждом MR и коммите в GitLab. Гарантирует, что новый код не ломает существующую функциональность. Зелёный CI — обязательное условие для мержа.
SLA
Соглашение об уровне обслуживания. Фиксирует: доступность системы (например, 99,9%), время реакции на инциденты по критичности, ответственных. Утверждается при передаче системы в эксплуатацию.

Состав команды ЭТП «ЦДТ». Роли соответствуют фазам процесса разработки — ответственный за каждую фазу указан в описании соответствующего этапа.
